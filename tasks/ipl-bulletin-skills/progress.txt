## Codebase Patterns
- Skills are loaded from a directory of `<name>/SKILL.md` files via `loadSkillFromDirectory(...)` in `packages/core/src/skills/`; a `handler: handler.ts` frontmatter field exposes the handler module as a scoped tool inside the child run, alongside any host tools listed in `allowedTools`.
- LLMs are unreliable simulators. Long Monte Carlo or numerical work should live in a deterministic handler (executable skill) and be invoked once via the synthetic `delegate.*` tool, not generated through model tokens. Concrete prior failure: 21-april-log.md records a 76-minute step-5 model timeout on the IPL run, almost certainly because the model was rolling dice in tokens.
- Public IPL data sources are unreliable enough that `examples/ipl2.sh` blacklists `espncricinfo.com` and `iplt20.com` directly in the prompt. Prefer a stable JSON mirror, configurable via env, with a date-pinned fixture for offline tests and CI.
- The `examples/skills/` directory is referenced by `examples/list-skills.ts` and `examples/README.md` but is not present in the current clone. US-001 creates it. Skills are not gitignored (.gitignore inspected), so they belong checked in.
- `examples/list-skills.ts` is the canonical loader sanity check; new skills should print correctly through it before they are wired into a goal.
- Acceptance criteria in this repo end with `Tests pass` and `Typecheck passes`. Mirror that discipline so each US is independently falsifiable.
- The repo root has no `vitest.config.ts`; run skill tests with `bunx vitest run <test-file>` from the repo root rather than from `packages/core` (the core's `include` glob does not reach into `examples/`).
- 2 preexisting test failures in `packages/core` reference `examples/skills/log-analyser/` which is missing from this clone; not in our scope to fix.

## 2026-04-30 - US-007
- Added `simulate-match.ts` implementing a Bradley-Terry style probability model.
- Pure function — no randomness — returns `{ pHome, pAway, pTie }` with constant 2% tie probability and the rest split via `sigmoid(strength_diff + home_advantage + form_diff)`.
- Default weights: `strength: 1.0`, `form: 0.4`, `home: 0.15`. All overridable per call.
- 7 tests pass covering symmetry (with home advantage zeroed), weight sensitivity, and intrinsic determinism. Total 34/34.
- **Learnings:**
  - Pure deterministic match probabilities must be computed *outside* the rollout loop; only outcome sampling needs the PRNG. This separation made the 10k-iteration performance budget trivial to hit.
  - Symmetry tests need to zero the home-advantage term so the swap actually inverts probabilities — otherwise the home bias shifts with the swap and the test reads as a model bug.

## 2026-04-30 - US-008
- Added `simulate-tournament.ts` running N iterations of the remainder.
- Each iteration replays remaining fixtures by sampling outcomes from `simulate-match` probabilities, updates a working points table, and records playoff/final/winner placement.
- Aggregates produce probabilities rounded to 4 decimals.
- Performance test confirms 10,000 iterations on 8 teams / 30 fixtures completes well under 2s on this machine. The actual IPL fixture (10 teams, 9 remaining) runs in ~40ms at 10k iterations.
- 7 tests pass; total 41/41.
- **Learnings:**
  - Pre-computing match probabilities once per tournament call (not once per iteration) cut simulation time roughly 10x. Probabilities are deterministic given inputs; only the outcome sample is random.
  - winnerProb sums to ~1.0, finalProb sums to ~2.0 (top-2), playoffProb sums to ~playoffSlots. These invariants are easy regression tests to encode and catch off-by-one bugs in the aggregation loop.

## Codebase Patterns (added on US-015 wrap-up)
- **When to push work into a deterministic handler vs. leave it to the LLM**: any computation that (a) has a numeric correctness contract, (b) requires sampling/iteration, or (c) is bottlenecked on token throughput should live in a `handler.ts` exposed as a `delegate.*` tool. The Monte Carlo simulation is the canonical example: 76 min in tokens vs 40 ms in TS, with deterministic and testable output. Reserve the LLM for tasks that genuinely need natural-language reasoning — orchestration, narrative styling, and fact-vs-claim arbitration.
